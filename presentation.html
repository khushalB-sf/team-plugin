The Problem

Why do you keep explaining the same thing?

Monday session

"When writing TypeScript, use zod for all
external data, never use any, use our
Result type for errors, follow our logging
pattern from lib/logger.ts..."

// Claude follows it. Great!

Wednesday (new session)

// Context cleared.
Claude writes raw try/catch.
Uses any types.
No logging.
No Result pattern.

"What happened to the patterns?"

Plugins solve this. They encode your standards once, and Claude follows them every session — without being asked.

Overview

A plugin is one package — with four building blocks inside

🎯

Skills

Claude auto-applies them when context matches. You never type anything — they just fire.

Example: every UI you build follows your design system automatically.

⚡

Commands

Slash commands you type to kick off a defined multi-step workflow.

Example: /commit — reads diff, matches commit style, creates commit.

🔌

MCP Servers

Gives Claude actual tool access — not just knowledge, but the ability to call APIs and run tools.

Example: Claude controls a real browser, creates GitHub issues, reads Linear.

🪝

Hooks

Shell commands that run automatically on Claude Code lifecycle events — no prompt needed.

Example: run a linter after every file edit, or notify Slack when a session ends.

# One plugin can have all four, or just one — your choice
commit-commands/          ← 3 Commands, no MCP, no Skills, no Hooks
playwright/               ← 1 MCP server, no Commands
team-plugin/              ← 1 Skill + 2 Commands, no MCP
your-plugin/              ← mix whatever you need (incl. Hooks)

Installation

Installing plugins

# User scope (default — just you)
/plugin install commit-commands@claude-plugins-official

# Local scope (just you, this project)
/plugin install commit-commands@claude-plugins-official \
  --scope local

# Project scope (whole team via git)
/plugin install commit-commands@claude-plugins-official \
  --scope project
git add .claude/
git commit -m "chore: add team plugins"

# List / update / remove
/plugin list
/plugin update
/plugin remove commit-commands@claude-plugins-official

Install scopes

Scope Flag Stored at Who gets it
User (default) --scope user ~/.claude/plugins/ Just you, all projects
Local --scope local ./.claude/plugins/ (gitignored) Just you, this project
Project --scope project ./.claude/plugins/ (commit to git) Whole team

Install sources

Source Example

Build Your Own

A plugin is just three files

team-plugin/
├── .claude-plugin/
│   └── plugin.json      ← name, description, author
├── skills/
│   └── team-standards/
│       └── SKILL.md    ← auto-triggered guidance
├── commands/
│   ├── standup.md      ← /standup
│   └── ticket.md       ← /ticket AUTH-234
└── hooks/
    └── post-edit.sh    ← runs after every file edit
# Install for the team (copy to project plugins dir)
mkdir -p .claude/plugins
cp -r team-plugin .claude/plugins/
git add .claude/ && git commit -m "chore: team plugin"

plugin.json

{
  "name": "team-plugin",
  "description": "Our standards",
  "author": {
    "name": "Simform Engineering"
  }
}

No build step. No compilation. Plain Markdown. Version-controlled like any code.

Build Your Own · Demo

team-plugin — purpose & usage

A plugin built for this repo. Encodes your team's coding standards and daily workflow shortcuts — so every engineer on the team gets the same behaviour without copy-pasting prompts.

Skill · Auto-triggered

team-standards

Fires automatically whenever Claude writes or edits TypeScript / JavaScript — no command needed.

Enforces:

  • Result pattern — no throw for domain errors
  • Structured logging on every I/O function
  • No any — Zod for external data
  • Promise.all() for parallel async calls

Command · You type it

/standup [hours]

When: Every morning before standup. Pass hours to look further back (default 24h).

What Claude does:

  • Reads git log for your commits
  • Reads git status for in-progress work
  • Formats Yesterday / Today / Blockers ready to paste
/standup
/standup 48   # last 48 hours

Command · You type it

/ticket <id>

When: Starting a new task. Pass your ticket ID — Claude creates a properly named branch.

What Claude does:

  • Reads ticket title from Linear / Jira
  • Creates kebab-case branch name
  • Checks out the new branch ready to code
/ticket AUTH-234
# → auth-234/add-refresh-token-rotation
mkdir -p .claude/plugins && cp -r team-plugin .claude/plugins/
Build Your Own · Real-World Tip

Debugging: /doctor after installing a custom plugin

Error from /doctor

Plugin team-plugin not found in marketplace
claude-plugins-official

Root cause

The plugin was registered in installed_plugins.json as team-plugin@claude-plugins-official — but was never added to the marketplace catalog. Claude Code validates every installed plugin against that catalog at load time.

The fix — two catalog files

# 1. Add entry to the catalog cache
~/.claude/plugins/plugin-catalog-cache.json
# ↳ add "team-plugin@claude-plugins-official": { ... }

# 2. Register in the marketplace manifest
~/.claude/plugins/marketplaces/claude-plugins-official/
  .claude-plugin/marketplace.json
# ↳ add { "name": "team-plugin", "source": "./plugins/team-plugin" }

# 3. Reload to verify — error gone
/reload-plugins
Reloaded: 2 plugins · 0 errors

How validation works

installed_plugins.json
tracks what's installed & where
↓ validates against
plugin-catalog-cache.json
server-fetched list of all marketplace plugins
↓ sourced from
marketplace.json
anthropics/claude-plugins-official on GitHub

Heads-up: The catalog cache refreshes from Anthropic's servers periodically. For a permanent fix, submit the plugin via PR to anthropics/claude-plugins-official, or install from a local path (not @marketplace) so it's never validated against the catalog.

Sharing Plugins

How to share plugins — local & global

Share with your team (project scope)

Install with --scope project — Claude writes the plugin into .claude/plugins/ inside your repo. Commit that folder and every teammate gets it on git pull.

# Copy plugin into project scope dir
mkdir -p .claude/plugins
cp -r team-plugin .claude/plugins/
git add .claude/
git commit -m "chore: add team-plugin"
git push
# teammates: git pull — plugin loads automatically

Private GitHub repo (team with access)

Push the plugin to a private GitHub repo first, then teammates clone it into their plugins directory.

# User scope — works across all projects
git clone https://github.com/<org>/<plugin-repo> \
  ~/.claude/plugins/<plugin-name>

# Project scope — commit to git
git clone https://github.com/<org>/<plugin-repo> \
  .claude/plugins/<plugin-name>

Share with everyone (global / public)

Public GitHub URL

Push to a public GitHub repo. Anyone can clone it into their plugins directory — no marketplace approval needed.

git clone https://github.com/<you>/my-plugin \
  ~/.claude/plugins/my-plugin

Official marketplace (anthropics/claude-plugins-official)

Submit a PR to the official repo. Once merged, anyone can install using the short @claude-plugins-official handle. 

# After PR is merged:
/plugin install my-plugin@claude-plugins-official
The Real Advantage

Plugins vs Normal Prompting

Aspect Normal Prompting Plugin
Activation You include it in every prompt Auto-fires when context matches
Persistence Lost when session ends Always active
Team consistency Each dev prompts differently Same standards for everyone
Maintenance Update every engineer's habit Update one file in git
Onboarding New devs don't know the prompts Install the plugin → instant context
External tools You explain the API each time Claude calls the API directly

"Normal prompting is a conversation.
Plugins are institutional memory."

The conversation ends. The plugin doesn't.
