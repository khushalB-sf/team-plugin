The Problem

Every new session, Claude starts from zero

Tuesday — deep debugging session

"Found the root cause: the race condition
in useAuth happens when token refresh
fires during route transition. Fixed it
by deferring the refresh to the onIdle
lifecycle hook."

// 2-hour investigation. Problem solved.

Wednesday (new session)

// Session context: zero.
"I see a useAuth hook.
What's the issue you're working on?"

// All investigation context: gone.
// Architecture decisions: gone.
// Two hours re-explained every time.

claude-mem gives Claude a persistent long-term memory — captured automatically, retrieved semantically, and injected intelligently at the start of each new session.

Purpose

What claude-mem actually does

It's a memory compression system that monitors every session in the background, records what Claude does and observes, compresses it semantically, and silently brings it back the next time you work on the same thing.

🧠

Capture

Hooks into Claude's tool lifecycle. Every file read, edit, and search generates an observation that gets stored automatically.

You do nothing. The hooks handle it.

🗜️

Compress

A worker process compresses raw observations into concise AI-generated summaries — filtering noise, preserving insight.

Token cost is amortized at write time, not read time.

🔍

Retrieve

Hybrid semantic + keyword search over a vector database (Chroma). Ask in natural language — it finds what's relevant.

Three-layer retrieval: index → IDs → full details.

💉

Inject

At session start, relevant memory is automatically injected into Claude's context — no prompting needed.

Progressive disclosure keeps token usage minimal.

Architecture

Five hooks + one worker = persistent context

Lifecycle hooks — what fires when

SessionStart → injects relevant past memory into context
UserPromptSubmit → records the user's intent
PostToolUse → stores tool result observations
Stop → triggers compression of session batch
SessionEnd → final summary & cleanup 
# Storage stack
SQLite            ← sessions, raw observations
Chroma (vectors)  ← semantic search index
Worker :37777     ← HTTP API + web viewer

Memory retrieval flow

STEP 1
Search index
Compact result — cheap tokens
↓
STEP 2
Identify relevant IDs
Filter to what's actually needed
↓
STEP 3
Fetch full details
Only for the IDs you chose

Progressive disclosure — never loads everything at once. Claimed 10× token savings vs. dumping full context.

Installation

Set up in one command

Install

# Claude Code (default)
npx claude-mem install

# Gemini CLI
npx claude-mem install --ide gemini-cli

# OpenCode
npx claude-mem install --ide opencode

# Via plugin marketplace
/plugin marketplace add thedotmack/claude-mem
/plugin install claude-mem

Requires Node.js 20+. Bun and Python uv (for Chroma) auto-install on first run.

Configuration — ~/.claude-mem/settings.json

{
  "model": "claude-sonnet-4-5",
  "workerPort": 37777,
  "dataDir": "~/.claude-mem/data",
  "logLevel": "info",
  "contextInjection": true,
  "CLAUDE_MEM_MODE": "code"
}

Mark sensitive content with <private>...</private> tags — claude-mem will skip capturing that content.

Web viewer available at http://localhost:37777 once the worker starts.

Usage

Querying your memory

mem-search skill (natural language)

# Find observations about auth bugs
search(
  query="authentication bug",
  type="bugfix",
  limit=10
)

# Get chronological context around event
timeline(observation_id=123)

# Fetch full observation details by IDs
get_observations(ids=[123, 456, 789])

# Generate a bug report from memory
npm run bug-report

Web viewer — localhost:37777

Real-time stream of observations as they're captured. Browse by session, search by keyword, inspect raw observations.

REST API

# Fetch a specific observation
GET localhost:37777/api/observation/123

# List sessions
GET localhost:37777/api/sessions

Automatic injection: you don't need to call search() manually. The SessionStart hook does it silently before your first message.

Pros

Where claude-mem shines

🤖

Fully automatic

Zero manual effort. Install once — hooks capture every session silently. No CLAUDE.md edits, no prompting discipline required.

💰

Token-efficient

Progressive disclosure (index → IDs → details) avoids loading the full history. Only relevant context is injected — up to 10× savings claimed.

🔍

Semantic search

Chroma vector DB means you can query memory in plain English — not just exact-string matching. "What did we decide about the token refresh?" just works.

🔒

Privacy controls

<private> tags give per-content exclusion. Credentials, PII, and sensitive notes stay out of the memory store entirely.

🌐

Multi-platform

Works with Claude Code, Gemini CLI, OpenCode, and OpenClaw. One memory store spans all your AI assistants.

🕰️

Timeline context

timeline() retrieves the chronological window around any observation — great for reconstructing what led to a decision.

Cons & Trade-offs

Know the limitations before adopting

Heavy dependency stack

Requires Node.js 20+, auto-installs Bun, and pulls in Python uv + Chroma for vector search. That's a significant footprint for a dev tool.

Local worker process required

A background worker on port 37777 must be running for memory to work. Adds one more long-running process to manage alongside your dev stack.

Beta / experimental features

Endless Mode and several other features are labelled beta. API shape and behaviour may change. Not production-hardened yet.

Windows PATH issues

Windows users need manual PATH adjustments for Bun and uv. First-run experience is rougher than on macOS/Linux.

Memory quality depends on AI compression

Summaries are generated by a model — they can mis-summarise or drop nuance. Garbage-in, garbage-out: if the compression model makes a mistake, that mistake is what gets retrieved later.

Disk & compute overhead

Every tool call writes an observation. Over weeks of use on large projects, SQLite + Chroma storage can grow substantially without manual pruning.

claude-mem vs Manual CLAUDE.md

Aspect Manual CLAUDE.md claude-mem

Hands-on · Steps 1–3

Scenario: debug a bug, come back tomorrow

1

Install claude-mem

npx claude-mem install

2

Trigger a realistic investigation

# In Claude Code, ask:
"The login form submits twice on mobile.
 Investigate and find the root cause in
 src/components/LoginForm.tsx"

# Claude reads files, greps the codebase,
# and makes edits — all captured as
# observations by PostToolUse hook.
3

End the session

# Close Claude Code (or type /exit)
# SessionEnd hook fires:
# observations → compressed → SQLite + Chroma

Check the web viewer at localhost:37777 to see captured observations.

4

Start a new session — watch auto-inject

# Open fresh Claude Code session.
# SessionStart hook fires silently.
# Claude already knows:
# - you were debugging LoginForm.tsx
# - the double-submit root cause
# - files read during investigation
# No prompting needed.

Query memory manually

# Ask Claude to search memory:
"Search memory for anything related

6

Verify via web UI

# Open browser
open http://localhost:37777

# Browse session · search observations
# GET /api/observation/42 for raw JSON

"The session ends.
The memory doesn't."

claude-mem turns every Claude Code session into a permanent, searchable record — so the next session starts where the last one left off.
